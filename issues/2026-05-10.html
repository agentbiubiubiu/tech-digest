🔥 头条深读

Agent 框架可以被 RCE 了。微软安全团队公开了 Semantic Kernel 中两类漏洞（CVE-2026-26030 / CVE-2026-25592）：攻击者通过提示注入操纵模型输出的工具参数，让框架把恶意内容喂给 Python eval 或文件 API，从而在 Agent 主机上实现远程代码执行或沙箱逃逸。关键洞察：漏洞不在模型，而在「框架如何信任解析后的工具数据」——你的 LLM 不是安全边界，你暴露的工具才定义了攻击面。

同一周，Agent 已经在替团队干活了。Linear 的客户案例显示 Ramp 内部 60% 的合并 PR 由编码 Agent 完成；Stripe 为 Agent 支付集成发布了可复现的评测基准（含全栈环境、确定性评分器与真实 Checkout 验证），强调支付场景「几乎正确即失败」；美团公开了 31 万行代码的 AI 重构实践，核心方法论是「用 Agent 评测思路约束 AI」——Rule 建设、Pre-PR 机制和分步评测比让 AI 写得更快重要得多。

监管侧也在跑步跟进。CISA 与多国网络安全机构联合发出代理型 AI 部署警示，敦促最小权限、身份治理与持续监控。CSA 发文指出 Agent 身份治理「被反着解」——先给了权限再想管控，修正窗口正在关闭。从技术漏洞到监管要求，Agent 安全已从概念讨论进入工程实战。

对团队的启发。Agent 不再是实验项目，而是在写代码、跑支付、做重构的生产系统。但它的安全模型远未成熟：工具调用缺乏输入校验、身份用共享服务账号、权限没有任务级限制。团队需要今天就回答三个问题：① 我们的 Agent 暴露了哪些工具，每个工具的输入白名单是什么？② Agent 用的凭据是共享的还是独立的？③ 出错时的回滚机制和审计日志在哪？不回答这些，就是在用生产环境做 Agent 安全的压测。

🧩 本期概念库

Agent Trust Boundary（Agent 信任边界）

解释：与传统安全边界不同，Agent 信任边界需要在「模型输出→工具调用→外部系统」的每个节点定义：哪些输出可信、工具接受什么格式的输入、权限范围多大。微软 RCE 案例证明，当框架对解析后的工具数据给予隐式信任，模型的一次幻觉就可能变成一次代码执行。
出处：微软安全博客 Semantic Kernel RCE 披露 + CSA Agent 身份治理讨论
重要性：Agent 安全从「提示注入导致输出偏差」升级到「提示注入导致远程代码执行」——这是一个质变。
团队可做：为每个 Agent 暴露的工具建立输入白名单、输出格式约束和最小权限策略，任何新工具接入都需要经过安全审查。

Eval Cost Explosion（评测成本爆炸）

解释：当评测从静态数据集进化到智能体交互、多轮验证、评分可靠性检验时，单次评测成本可能从几美元跃升至数千美元。Hugging Face 用 HAL、PaperBench 等实际账本展示：可靠性验证（多轮重复 + 评委一致性）进一步放大账单。
出处：Hugging Face Blog 'AI evals are becoming the new compute bottleneck'
重要性：评测不再是免费的事后检查——它有自己的 FinOps 模型。不计入预算的评测会在 Agent 上线后产生意外成本。
团队可做：在 Agent 工具链预算中预设「评测预算」行项，并在模型切换/升级时纳入评测成本变化的评估。

Important Decisions Document（IDD，重要决策文档）

解释：ACM Queue 提出的轻量治理实践：为长期项目记录关键决策——做了什么、为什么、否定了哪些替代方案。目的是减少反复争论、帮助新人快速理解上下文、防止「为什么当时这么做」的组织性遗忘。
出处：ACM Queue 'The Important Decisions Document'
重要性：当 AI 加速了代码生成速度，架构决策的半衰期也在缩短——记录「为什么」比记录「是什么」更持久。
团队可做：为跨端交易平台的核心技术选型（如架构迁移、依赖替换、Agent 工具链选择）建立 IDD 实践，每个重大决策附带 2-3 个被否定的替代方案说明。
